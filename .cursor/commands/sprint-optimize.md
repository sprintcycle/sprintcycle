# SprintCycle 系统优化工作流

## 适用范围

本工作流适用于以下**中等以上优化需求**：

| 需求类型 | 复杂度 | 是否适用 |
|---------|--------|----------|
| 字段整合/删减 | 中 | ✅ |
| DDD 架构治理 | 中 | ✅ |
| 兼容逻辑删除 | 中 | ✅ |
| 前后端契约对齐 | 中 | ✅ |
| 核心模块重构 | 高 | ✅ |
| 架构演进 | 高 | ✅ |
| 性能优化 | 中 | ✅ |
| 安全加固 | 中 | ✅ |

**简单修复（如单文件bug fix）** 可直接处理，无需走此工作流。

## 工作流触发

### 触发方式

1. **命令触发**（推荐）
   ```
   /sprint-optimize
   ```

2. **触发词识别**
   - 「删减字段」
   - 「遵循 DDD 治理」
   - 「删除兼容逻辑」
   - 「优化架构」
   - 「精简代码」
   - 「前后端对齐」
   - 「重构」
   - 「性能优化」
   - 「安全加固」

## 工作流阶段

```
┌─────────────────────────────────────────────────────────────────────┐
│                      SprintCycle 优化工作流                          │
├─────────────────────────────────────────────────────────────────────┤
│  Phase 1: 需求分析 → Phase 2: 方案设计 → Phase 3: 执行实施          │
│         ↓                    ↓                    ↓                 │
│  Phase 4: 测试验证 → Phase 5: 文档同步 → Phase 6: 审批提交          │
└─────────────────────────────────────────────────────────────────────┘
```

## Phase 1: 需求分析（必须）

### 1.1 理解业务背景

- [ ] 与需求提出者确认优化目标和预期收益
- [ ] 分析现有代码结构和业务逻辑
- [ ] 识别受影响的模块和调用链

### 1.2 影响范围评估

| 评估维度 | 检查项 | 风险等级 |
|---------|--------|----------|
| **业务影响** | 是否影响核心业务流程？ | 高/中/低 |
| **代码范围** | 涉及多少文件/模块？ | 高/中/低 |
| **前后端关联** | 是否需要同步修改前端？ | 是/否 |
| **外部依赖** | 是否影响外部系统集成？ | 是/否 |
| **测试覆盖** | 现有测试是否覆盖变更点？ | 是/部分/否 |

### 1.3 输出文档

- **优化需求分析报告**（包含：问题描述、影响范围、风险评估）

## Phase 2: 方案设计（必须）

### 2.1 架构合规性检查

对照 [sprintcycle-architecture-orchestration.mdc](file:///Users/liangzai/CursorProjects/sprintcycle/.cursor/rules/sprintcycle-architecture-orchestration.mdc)：

- [ ] 符合 DDD 六边形架构原则
- [ ] 保持领域层纯粹性
- [ ] 遵循组合根模式
- [ ] 保持聚合根不可变性

### 2.2 方案设计

根据优化类型选择方案模板：

#### 方案类型 A：字段整合

1. 识别语义相关的字段组
2. 设计统一的上下文结构
3. 列出所有调用点（前端 + 后端）
4. 制定迁移计划

#### 方案类型 B：DDD 治理

1. 识别架构违规模式
2. 设计合规的改进方案
3. 确保边界清晰
4. 制定重构策略

#### 方案类型 C：兼容逻辑删除

1. 定位兼容辅助方法
2. 识别所有调用点
3. 制定移除计划

#### 方案类型 D：前后端对齐

1. 检查后端 DTO 定义
2. 验证前端 API/类型/Store
3. 制定同步更新计划

| 模块 | 文件路径 | 变更类型 | 负责人 | 预计工时 |
|------|---------|----------|--------|----------|

| 后端应用层 | `sprintcycle/application/services/` | 修改/新增/删除 | | |

| 前端视图 | `frontend/src/views/` | 修改/新增/删除 | | |

### 2.4 输出文档

- **优化方案设计文档**（包含：架构图、变更清单、迁移计划）

## Phase 3: 执行实施（必须）

### 3.1 后端变更流程

```
领域层 → 应用层 → DTO → HTTP路由 → 测试
```

1. **领域层**：修改聚合根、值对象、领域服务
   - 确保 `@dataclass(frozen=True)` 不可变性
   - 更新 `domain/ports/` 端口定义

2. **应用层**：更新服务实现和 DTO
   - 保持组合根纯净性
   - 确保 DTO 与前端类型对齐

3. **接口层**：更新 HTTP 路由和 handlers
   - 使用一致的命名约定
   - 保持 API 版本兼容性

### 3.2 前端变更流程

```
API定义 → TypeScript类型 → Store → 视图组件 → 验证
```

1. **API 层**：更新 `frontend/src/api/` 接口定义
   - 与后端 DTO 保持一致

2. **类型层**：更新 `frontend/src/types/` 类型定义
   - 镜像后端领域模型

3. **Store 层**：更新状态管理
   - 与后端聚合根对齐

4. **视图层**：更新组件
   - 确保数据展示正确

### 3.3 变更原则

- ✅ 小步增量修改
- ✅ 保持可回滚性
- ✅ 优先更新测试
- ✅ 前后端同步变更

## Phase 4: 测试验证（必须）

### 4.1 后端测试

- [ ] 运行相关 pytest 测试
- [ ] 确保所有测试通过
- [ ] 验证业务逻辑完整性

### 4.2 前端测试

- [ ] 运行 TypeScript 类型检查
- [ ] 运行 Playwright 端到端测试（如有）
- [ ] 手动验证关键功能

### 4.3 集成测试

- [ ] 验证前后端接口通信
- [ ] 验证数据流完整性
- [ ] 验证错误处理机制

### 4.4 性能测试（如适用）

- [ ] 基准性能测试
- [ ] 对比优化前后指标
- [ ] 确保性能达标

## Phase 6: 审批提交（必须）

### 6.1 代码审查

- [ ] 提交 Pull Request
- [ ] 指定至少 2 位 reviewer
- [ ] 确保代码符合架构规范
- [ ] 确保测试覆盖率达标

### 6.2 审批流程

| 审批环节 | 责任人 | 审批内容 |
|---------|--------|----------|
| 技术负责人 | 架构师 | 架构合规性 |
| 业务负责人 | 产品经理 | 业务逻辑正确性 |
| QA负责人 | 测试主管 | 测试覆盖完整性 |

### 6.3 提交规范

**提交信息格式：**
```
[优化类型] 简明描述变更内容

详细描述：
- 变更原因
- 主要修改
- 影响范围

验证结果：
- 后端测试：✅/❌
- 前端测试：✅/❌
- 文档更新：✅/❌
```

**示例：**
```
[字段整合] 整合 lifecycle 中 skill 相关字段

详细描述：
- 将 skill_refs、skill_matches、skill_review_checklists、skill_trace 整合为 skill_context
- 更新领域模型、DTO、HTTP 路由
- 同步更新前端 API、类型和 Store

验证结果：
- 后端测试：✅ 全部通过
- 前端测试：✅ 全部通过
- 文档更新：✅ README.md、ARCHITECTURE_INVARIANTS.md
```

## 核心原则

### 必须遵守

- ✅ 保持业务逻辑完整
- ✅ 遵循 DDD + 六边形架构
- ✅ 小步增量修改
- ✅ 测试全部通过
- ✅ 前后端同步变更
- ✅ 文档及时更新

### 禁止行为

- ❌ 不要破坏现有功能
- ❌ 不要引入架构违规
- ❌ 不要删除未验证的逻辑
- ❌ 不要忽略测试失败
- ❌ 不要前后端变更不同步
- ❌ 不要跳过审批流程
