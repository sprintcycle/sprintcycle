# SprintCycle 架构优化命令

当你需要优化 SprintCycle 代码时使用此命令。

## 触发词

- 「删减字段」
- 「遵循 DDD 治理」
- 「优化架构」
- 「删除兼容逻辑」
- 「精简代码」

## 执行流程

### Step 1 - 分析阶段

1. **理解现有架构**
   - 阅读相关代码文件和现有模式
   - 理解业务逻辑和数据流
   - 识别冗余字段、重复逻辑、兼容性代码

2. **识别优化点**
   - 字段整合（合并冗余字段）
   - 架构合规（遵循 DDD + 六边形）
   - 代码简化（删除兼容逻辑）
   - 逻辑保持（验证业务逻辑不丢失）

### Step 2 - 方案制定

根据优化类型选择方案：

**字段删减场景：**
1. 识别需要合并的字段组
2. 设计新的统一字段结构
3. 列出所有调用点
4. 制定迁移计划

**架构治理场景：**
1. 对照 `.cursor/rules/sprintcycle-architecture-orchestration.mdc`
2. 识别违规的代码模式
3. 设计符合架构的改进方案
4. 确保 DDD 边界清晰

**兼容逻辑删除场景：**
1. 找到所有兼容辅助方法
2. 识别所有调用点
3. 验证测试覆盖
4. 移除兼容代码

### Step 3 - 代码落地

1. **小步快跑**
   - 先改模型/核心代码
   - 更新服务层调用
   - 验证编译通过

2. **测试验证**
   - 运行相关测试
   - 确保所有测试通过
   - 验证业务逻辑完整

3. **提交代码**
   - 编写清晰的提交信息
   - 列出所有变更文件
   - 确保变更可追溯

### Step 4 - 文档同步（强制）

⚠️ **代码变更后必须立即更新文档！**

更新以下 4 个文档文件（以代码实现为准）：

1. **README.md**（中文）
   - 更新产品定义和能力矩阵
   - 同步字段结构和架构描述
   - 反映新的 DDD 模式和生命周期阶段

2. **README_EN.md**（英文）
   - 镜像 README.md 的所有变更
   - 保持双语一致性

3. **docs/ARCHITECTURE_INVARIANTS.md**
   - 更新架构边界和不变性
   - 同步 DDD 模式（聚合根、值对象）
   - 反映 Port/Adapter 映射
   - 更新层职责

4. **.cursor/rules/sprintcycle-architecture-orchestration.mdc**
   - 与实际代码实现对齐
   - 更新 DDD 子域结构
   - 同步聚合根定义
   - 反映新模式和约定

**文档更新原则：**
- ✅ 以代码实现为准
- ✅ 保持双语一致性
- ✅ 文档间无矛盾
- ✅ 反映真实架构

**文档更新检查点：**
- [ ] README.md 字段描述与代码一致
- [ ] README_EN.md 与 README.md 对齐
- [ ] ARCHITECTURE_INVARIANTS.md 边界与代码一致
- [ ] sprintcycle-architecture-orchestration.mdc 与代码实现对齐

## 核心原则

### 必须遵守

- ✅ 保持业务逻辑完整
- ✅ 遵循 DDD + 六边形架构
- ✅ 字段整合要有语义
- ✅ 小步增量修改
- ✅ 测试全部通过

### 禁止行为

- ❌ 不要破坏现有功能
- ❌ 不要引入架构违规
- ❌ 不要删除未验证的逻辑
- ❌ 不要忽略测试失败

## 验证清单

- [ ] 代码编译通过
- [ ] 相关测试全部通过
- [ ] 业务逻辑验证完整
- [ ] 架构边界检查合规
- [ ] 提交信息清晰
