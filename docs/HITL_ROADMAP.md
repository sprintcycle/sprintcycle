# SprintCycle HITL 后续增强路线图

## Phase 1 现在就能补的增强

### 1. 更完整的 `modify` 语义
当前已经能表达 `request_changes`，但还可以增强为：

- 返回修改建议模板
- 记录修改原因
- 允许把修改建议带回 spec / release plan / task context
- 修改后重新进入 pending

### 适用场景
- Spec 需要调整
- 任务执行方案需要修订
- Release Plan 需要重审

### 2. `retry` 语义
增加更明确的重试动作：

- 重新生成 spec
- 重新跑 planning gate
- 重新发起 review
- 重新执行某个 sprint 或 task

### 好处
比单纯的 `approve/reject` 更符合实际人工介入场景。

### 3. HITL 审计视图
补一个统一审计视图，展示：

- 谁触发了 HITL
- 哪个 gate 触发的
- 什么时候创建
- 谁做了决策
- 决策备注是什么
- 是否超时
- 是否重试

## Phase 2 中期增强

### 4. 多审批人机制
支持更复杂的审批策略：

- 单人审批
- 双人审批
- N 人通过
- 角色审批
- 高风险 gate 强制升级审批

### 典型适用
- 生产发布
- 数据迁移
- 安全敏感改动
- 架构决策变更

### 5. 审批链路
从“单点审批”升级为“链式审批”。

例如：
- 先由技术负责人确认
- 再由产品负责人确认
- 最后由系统继续执行

### 6. 超时升级
当前 timeout 只是做默认决策，未来可以升级成：

- 超时自动提醒
- 超时升级到更高角色
- 超时转人工通知系统
- 超时进入更保守执行策略

## Phase 3 长期增强

### 7. 工作流引擎化
如果后续 HITL 场景非常复杂，建议考虑引入工作流引擎，例如 Temporal。

适合的情况：
- 长事务审批
- 多步骤确认
- 多角色参与
- 超时 / 重试 / 恢复 / 回放需求强

### 8. 外部通知通道
HITL 触发后可通过更多渠道通知：

- Slack
- 飞书
- 企业微信
- 邮件
- Webhook

### 9. 统一审批中心
最终可以演进成一个独立的审批中心服务：

- 统一存储
- 统一审计
- 统一通知
- 多项目共享
- 多用户协作

## 推荐优先级

如果按投入产出比排序，我建议：

### 优先做
1. `modify` 语义增强
2. HITL 审计视图
3. 超时升级
4. 多审批人机制

### 后续再做
5. 审批链路
6. 外部通知通道
7. 工作流引擎化
8. 独立审批中心

## 最终建议

当前 SprintCycle HITL 已经完成“能用、能接、能管”的阶段。下一阶段重点是把它从“可用”继续打磨成：

- 更强的审批语义
- 更完善的审计
- 更好的组织协作
- 更可靠的长事务能力
